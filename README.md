**A minimal experiment testing whether shorthand compression grounded in real
stenography makes small language models more efficient to train and run.**

Full build/iteration history (every version tried, not just the current one):
[`docs/log.md`](docs/log.md).

## 1. Summary

Human stenographers compress speech into a few keystrokes using systems like
Gregg, Pitman, and modern machine-steno "chords." GlyphLM asks a narrow, testable
question: if you compress written text the same way *before* it reaches a
language model's tokenizer, does the model train faster, run faster, and model
the language at least as well — at equal vocabulary budget?

Two identical tiny GPT-2 models are trained on the same corpus
(`karpathy/tiny_shakespeare`): one on the raw text, one on a version where the
most frequent words and phrases have been replaced by their real steno chords
from [Plover](https://github.com/openstenoproject/plover), an open-source
stenography engine. Everything else — architecture, hyperparameters, vocab size,
training steps — is held constant between the two.

## 2. Hypothesis

At an equal token budget, a model trained on shorthand-compressed text achieves:
- a better tokens-per-character compression ratio,
- comparable or better perplexity, and
- faster inference (tokens/second),

compared to an identical model trained on raw text.

Framed more precisely: any gap this experiment finds should tell us whether it's
a **fixed vocab-budget artifact** (novel shorthand symbols losing out to common
English morphemes when a BPE tokenizer has a limited number of vocab slots to
hand out) **or a genuine structural problem** with shorthand substitution itself.
That distinction — not just the raw numbers — is the point of running this as a
controlled comparison rather than reasoning about it in the abstract.

## 3. Method

| Component | What it does | File |
|---|---|---|
| Steno dictionary | Downloads Plover's real chord dictionary, filters it to plain word/phrase translations, ranks entries by frequency in the training corpus | `glyph/steno_dict.py` |
| Encoder | Fits a word/phrase → chord mapping from the corpus (top 150 words, top 30 phrases by frequency), then substitutes them via a single-pass regex | `glyph/encoder.py` |
| Data pipeline | Downloads `tiny_shakespeare`, produces a raw and a shorthand-encoded corpus, holds out a validation split (never used for training) | `glyph/data.py` |
| Tokenizer | Trains two independent BPE tokenizers, vocab_size=2048 — one per corpus. Shorthand chords are registered as BPE special tokens so they don't have to *compete* for merge slots against ordinary English subwords | `glyph/tokenizer.py` |
| Model | Two identical tiny GPT-2s (n_embd=128, n_layer=4, n_head=4, n_positions=256), 3 epochs, AdamW lr=3e-4, MPS/CUDA/CPU auto-detect | `glyph/train.py` |
| Evaluation | Held-out perplexity, inference tokens/sec, compression ratio, sample completions | `glyph/eval.py` |
| Chat | Interactive REPL: your input is shorthand-encoded, generated by the shorthand model, then decoded back to English | `glyph/chat.py` |

**Why this design is a fair comparison:** both models see the same underlying
information, the same vocab budget (2048 tokens), and the same architecture and
training budget. The only variable is whether common words/phrases have been
pre-compressed into steno chords before tokenization.

**Metric definitions** (as computed in `glyph/eval.py`):
- *Perplexity* — `exp(mean cross-entropy loss)` on the held-out validation split, computed per-token *in each model's own tokenization*.
- *Compression ratio* — BPE tokens ÷ character count of the (raw or shorthand) text.
- *Whitespace-token ratio* — glyph corpus word count ÷ raw corpus word count, via a plain whitespace split. This is the only metric that's directly comparable across corpora regardless of tokenizer, since it doesn't depend on BPE at all.
- *Tokens/second* — `model.generate()` throughput on a fixed prompt, greedy decoding.

## 4. How to run

```
uv venv --python 3.12 && uv pip install -r requirements.txt
python -m glyph.data && python -m glyph.tokenizer && python -m glyph.train
python -m glyph.eval
```

(Or open `GlyphLM_experiment.ipynb` locally or in Google Colab.) Chat with the
trained shorthand model: `python -m glyph.chat`.

Apple M5 MacBook Air (MPS backend), full pipeline in under a minute of compute —

converged, statistically significant experiment. See §7 for what that limits us

## 6. Discussion

This is the first iteration where the shorthand model wins on *every* metric —
lower loss, lower perplexity, faster inference — and the whitespace-token ratio
finally dropped below 1.0, meaning the encoder is genuinely reducing word count
via phrase-chording, not just shrinking character count. That matters because
whitespace-token ratio is the one metric in this table not entangled with either
model's own tokenizer, so it's the closest thing here to a tokenizer-independent
signal that the compression is real.

Two earlier iterations (`docs/log.md`) showed *why* this took three tries to get
right:
1. A hand-picked list of 10 words + suffix rules compressed characters but never
   reduced word count (substitution was always 1-word-in, 1-word-out), and its
   novel symbols actually *hurt* BPE compression by losing the frequency-based
   merge competition against common English morphemes.
2. Adding hand-picked phrase rules and reserving special tokens fixed perplexity
   dramatically, but made compression *worse* — reserving vocab slots for guessed
   symbols starved the BPE learner of merge budget elsewhere.
3. Replacing hand-picked guesses with real, frequency-ranked Plover steno chords
   (this run) improved every metric simultaneously, including narrowing the
   compression regression from iteration 2 by more than half.

The throughline: **the vocab budget is the real constraint**, not the idea of
shorthand compression itself. Iteration 2's regression and iteration 3's recovery
are both consistent with the "fixed vocab-budget artifact" framing in §2, not
with shorthand substitution being structurally unworkable.

**Chat example** (`python -m glyph.chat`, current `model_glyph`):
```
you> hi
glyph> hi S AEU EU TPOT , . C U . WE . E T . C : PWU R E : PHO ly ' PHEU HRAEUF , -T S PHEU TPHOEBL SR P-R T E ' s , R E E ' s : L E G I I s . -T , U , . TPOT s y US ;
plain> hi is a i not , . C you . we . he it . C : but are he : no ly ' my life , the is my noble have for it he ' s , are he he ' s : L he G I I s . the , you , . not s y us ;
```
The decode is faithful — `-T`→"the", `S`→"is", `SR`→"have", `P-R`→"for",
`TPOT`→"not" all resolve correctly via the fitted chord map — but the output
itself is still gibberish. That's expected: this is a ~4M-parameter model given
~11 seconds of training on 1MB of text. This example demonstrates the
encode → generate → decode pipeline is wired correctly, not that the model is
useful for conversation yet.

## 7. Limitations

- **Not a converged experiment.** Every run logged so far is a ~1-minute smoke
  test (~100 training steps). Loss is still far from a plateau for either model.
  Conclusions here are about whether the *mechanism* works, not about the
  asymptotic quality gap between raw and shorthand training.
- **Perplexity is compared per-token across two different tokenizations, which
  isn't strictly apples-to-apples** — a shorthand token can carry more original
  information than a raw token, so a lower per-token perplexity doesn't
  automatically mean "better language modeling per unit of real content." The
  correct normalization is bits-per-original-character (total cross-entropy in
  bits ÷ length of the *uncompressed* reference text); not yet implemented.
- **Single corpus, single seed, single vocab size.** All results are from one
  1MB corpus, one run each (no repeated seeds for variance), and one fixed
  vocab_size=2048.

## 8. Ideas for the next iteration

- Implement the bits-per-original-character metric in `eval.py` (§7).
- Vocab size sweep — since the remaining compression gap traces to a fixed
  2048-slot budget, trying a few sizes would show whether it closes, stays flat,
  or grows.
- Try SentencePiece Unigram instead of BPE — tends to compress slightly better
  at small vocab sizes.
- Train to convergence (more steps, more data) rather than a smoke test.

## 9. Related work

[Training LLMs over Neurally Compressed Text](https://arxiv.org/abs/2404.03626)
(Lester et al., 2024) trains LLMs directly on text compressed with a *learned*
neural compressor (Equal-Info Windows over arithmetic coding), showing this can
outperform byte-level baselines at scale — but also that naive/opaque
compression can be hard for a model to learn from at all, which is why they
needed Equal-Info Windows rather than raw arithmetic coding. GlyphLM's shorthand
substitution is much milder and fully human-interpretable by comparison — a
lightweight, differently-motivated complement to that direction rather than a
competing method at the same scale.

[Plover](https://github.com/openstenoproject/plover) (GPLv2+) supplies the real
steno chord dictionary this project's encoder is fitted from; its dictionary
data is fetched at build time and is not redistributed in this repository.
